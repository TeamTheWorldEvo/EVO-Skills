# Step 3：交互与流程规则

这三条是本项目实践中提炼的补充规则。逐条介绍，逐条确认。

## 3.1 批判性审查（Critical Review）

**解释：**

> **规则：批判性审查**
>
> **它做什么**：当你（用户）在 vibe coding 中提出建议时，AI 必须带着批判性眼光审查，不能盲从。如果 AI 认为你的建议有技术问题，必须明确指出、提出替代方案、与你讨论达成共识后再执行。
>
> **为什么需要它**：AI 编程中一个常见问题是 AI "讨好用户"——明知用户的指令会引发问题也照做。这条规则把 AI 定位为"技术合伙人"而非"代码打字员"。
>
> **开着会怎样**：AI 会更频繁地质疑你的决策，可能会让你觉得"啰嗦"。但能避免盲从导致的错误。
>
> **关着会怎样**：AI 默认你说什么就做什么，交互更快，但你需要自己对所有技术决策负责。
>

> - **B) 不要** — 我说了算，AI 照做就行

## 3.2 上下文地图更新（Context Map Update）

**解释：**

> **规则：上下文地图更新**
>
> **它做什么**：每次对话结束时，AI 会把这次对话中对项目的**非显而易见的新理解**（设计意图、模块关系、避坑记录）追加到 CLAUDE.md 的 `Context Map` 章节。
>
> **为什么需要它**：Claude Code 每次新对话是空白状态（除非用 /resume）。Context Map 是跨 session 的"记忆锚点"——下次对话时，AI 读到这些记录就能快速进入状态。
>
> **和 memory 系统的区别**：memory 存的是"规则和约束"；Context Map 存的是"这个项目具体发生了什么"。
>
> **开着会怎样**：CLAUDE.md 的 Context Map 会随时间自然增长。需要偶尔清理过时条目。
>
> **关着会怎样**：每次新对话 AI 需要重新理解项目上下文，可能重复问同样的问题。
>
> ---
>
> 要这条规则吗？
> - **A) 要（推荐）** — 让 AI 记住项目演进过程
> - **B) 不要** — 保持 CLAUDE.md 简洁

等待回答。记录选择。

## 3.3 工作目录约束（Work Directory Constraint）

**解释：**

> **规则：工作目录约束**
>
> **它做什么**：AI 的所有文件操作（读、写、改、搜索）**仅限本项目根目录及子文件夹内**。
>
> **为什么需要它**：防止 AI 意外修改用户的其他项目文件或系统配置。这是一个安全边界，不是功能限制。
>
> **注意**：这是**行为规则**（写在 CLAUDE.md 里的文字约束），和后面的**权限配置**（settings.json 里的硬限制）不同。行为规则靠 AI 自觉遵守；权限配置是系统级强制。两者配合效果最好。
>

>
> 要这条规则吗？
> - **A) 要（推荐）** — 双重保险
> - **B) 不要** — 靠后面的权限配置就够了
